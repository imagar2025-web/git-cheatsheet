1. Qué es Git y cómo funciona

Git tiene 3 zonas por las que pasa tu código antes de llegar a GitHub:

zona 1
Working Dir
Tus archivos
→
zona 2
Staging
git add
→
zona 3
Local Repo
git commit
→
remoto
GitHub
git push

Working Directory es tu carpeta con archivos. Staging es la "caja" donde preparas lo que quieres guardar. Commit es la foto guardada de tu código. Push sube esa foto a GitHub.

2. Los comandos que usamos para tu repo

Estos son exactamente los comandos que ejecutaste, explicados uno a uno:

$ git init

Crea un repositorio Git vacío en la carpeta actual. Genera una carpeta oculta .git/ que es donde Git guarda todo su historial. Solo se hace una vez por proyecto.

$ git add .

El punto . significa "todo". Añade todos los archivos de la carpeta al staging. Es como decir "quiero incluir todo esto en mi próximo commit". También puedes añadir archivos individuales: git add README.md.

$ git commit -m "feat: initial structure with exercises and projects"

Guarda una "foto" de todo lo que está en staging. El -m permite escribir el mensaje directamente. Sin -m se abre un editor de texto.

$ git branch -M main

Renombra la rama actual a main. Git por defecto crea una rama llamada master, pero el estándar actual es main. La -M (mayúscula) fuerza el renombrado.

$ git remote add origin https://github.com/imagar2025-web/daw-markup-and-web.git

Conecta tu repo local con el de GitHub. origin es el nombre que le damos al remoto (es una convención, podría ser cualquier nombre). Solo se hace una vez.

$ git push -u origin main

Sube los commits locales a GitHub. El -u establece que a partir de ahora, git push sin argumentos irá a origin/main automáticamente. Solo necesitas -u la primera vez.

Resumen

Después de la configuración inicial, tu flujo diario será solo: git add . → git commit -m "mensaje" → git push

3. El día a día con Git

Ver el estado actual

$ git status # Te dice qué archivos han cambiado, cuáles están en staging y cuáles no

Ver qué has cambiado

$ git diff # Muestra línea a línea lo que ha cambiado respecto al último commit

Ver historial de commits

$ git log --oneline # Lista compacta de commits: hash + mensaje a1b2c3d feat: add login form validation e4f5g6h feat: initial structure

Bajar cambios de GitHub

$ git pull # Descarga los commits nuevos del remoto y los fusiona con tu local
Regla de oro

Siempre haz git pull antes de empezar a trabajar si alguien más toca el repo (o si tú editas desde GitHub web).

4. Ramas: la clave del trabajo en equipo

Las ramas permiten trabajar en funcionalidades nuevas sin tocar el código estable. Piensa en ellas como líneas paralelas de desarrollo:

main ●───●───●───────────●───● \ / develop ●───●───●───● \ / feature ●───●

Crear y cambiar de rama

$ git branch # Lista todas las ramas (* marca la actual) $ git branch develop # Crea una rama nueva llamada "develop" $ git checkout develop # Cambia a la rama "develop" $ git checkout -b feature/login # Atajo: crea Y cambia a la rama en un solo comando

Flujo típico con ramas

# 1. Estás en main, creas rama para nueva funcionalidad $ git checkout -b feature/carrito-compras # 2. Trabajas, haces commits... $ git add . $ git commit -m "feat: add shopping cart component" $ git commit -m "feat: add item quantity selector" # 3. Cuando terminas, vuelves a main y fusionas $ git checkout main $ git merge feature/carrito-compras # 4. Subes a GitHub $ git push # 5. (Opcional) Borras la rama que ya no necesitas $ git branch -d feature/carrito-compras

Estrategia de ramas recomendada

RamaPara quéEjemplo
mainCódigo estable, siempre funcionalLo que está en producción
developDesarrollo activo (opcional para repos pequeños)Siguiente versión
feature/*Funcionalidades nuevasfeature/dark-mode
fix/*Corrección de bugsfix/login-redirect
hotfix/*Arreglos urgentes en producciónhotfix/security-patch
Para tus repos de DAW

No necesitas ramas complejas. Con main es suficiente para repos de ejercicios. Usa ramas solo en proyectos grandes (TFG, apps completas).

5. Resolver conflictos

Exactamente lo que te pasó al subir tu repo. Un conflicto ocurre cuando Git no sabe cuál de dos versiones de un archivo mantener.

Cómo se ve un conflicto

<<<<<<< HEAD Tu versión local del código ======= La versión del remoto (GitHub) >>>>>>> origin/main

Cómo resolverlo

$ git checkout --ours README.md # Usa tu versión local
$ git checkout --theirs README.md # Usa la versión de GitHub
# Abres el archivo, borras los marcadores <<<<< ===== >>>>> # y dejas el código que quieras. Luego: $ git add README.md $ git commit -m "fix: resolve merge conflict in README"

6. Conventional Commits

Los mensajes de commit siguen un formato estándar que hace el historial legible:

PrefijoCuándoEjemplo
feat:Nueva funcionalidadfeat: add user registration form
fix:Arreglar un bugfix: correct email validation regex
docs:Solo documentacióndocs: add API usage examples
style:Formato, no lógicastyle: fix indentation in main.css
refactor:Reestructurar sin cambiar funcionalidadrefactor: extract helper methods
chore:Tareas de mantenimientochore: update .gitignore
Mensajes que NO debes escribir

git commit -m "cambios" — ¿qué cambios?
git commit -m "asdf" — un reclutador verá esto
git commit -m "fix fix fix" — caos

7. Comandos de rescate

Deshacer el último commit (sin perder cambios)

$ git reset --soft HEAD~1 # Deshace el commit pero mantiene los archivos en staging

Corregir el mensaje del último commit

$ git commit --amend -m "feat: mensaje corregido" # Solo si NO has hecho push todavía

Descartar cambios de un archivo

$ git checkout -- archivo.html # Vuelve a la versión del último commit. ¡Cuidado, pierde cambios!

Sacar un archivo del staging

$ git reset archivo.html # Lo quita del staging pero mantiene los cambios en el archivo

Nuclear: volver a un commit anterior

$ git log --oneline a1b2c3d último commit e4f5g6h el commit bueno ← quiero volver aquí $ git reset --hard e4f5g6h # ⚠️ BORRA todo lo posterior. Úsalo solo si estás seguro.
Cuidado

git reset --hard es irreversible. Pierde código. Úsalo solo cuando sepas lo que haces.
